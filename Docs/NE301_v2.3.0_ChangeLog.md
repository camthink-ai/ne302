## ⚠️ BREAKING CHANGES

### 1. Flash Partition Table Redesign

The Flash partition layout has been restructured. **The new Web firmware must be re-uploaded**, and the system will prompt users when FSBL or WiFi firmware versions are mismatched.

| Partition  | Old (v2.1.0)           | New (v2.3.0)            | Change                                |
|-----------|------------------------|-------------------------|---------------------------------------|
| AI_1      | 5MB @ 0x70E00000       | 8MB @ 0x70900000        | Address moved forward, size doubled (merged AI_DEFAULT) |
| AI_2      | 5MB @ 0x71300000       | 8MB @ 0x71100000        | Address moved forward, size doubled (merged AI_3) |
| AI_DEFAULT | 5MB @ 0x70900000      | — Removed —             | Merged into AI_1                      |
| AI_3      | 8MB @ 0x71800000       | — Removed —             | Merged into AI_2                      |
| WEB       | — N/A —               | 1MB @ 0x71900000         | **New** independent Web partition      |
| WiFi FW   | 4MB @ 0x77C00000       | 3MB @ 0x71A00000        | Address moved forward, size adjusted   |
| LittleFS  | 64MB @ 0x72000000      | 32MB/96MB @ 0x71D00000  | Address changed, 64M/128M Flash support |
| Reserve2  | 28MB @ 0x76000000      | 4MB @ 0x73D00000/0x77D00000 | Address and size changed          |

### 2. PSRAM Memory Layout Changes

- **APP PSRAM boot**: The previous release already supported PSRAM execution via `BOOT_IN_PSRAM`. This release **removes the conditional compilation** — PSRAM boot is now the only mode. APP available space increased from 4032K to 4096K.
- **PSRAM SWAP region removed**: The 64KB PSRAM SWAP region at `0x903F0000` has been removed.
- **New SRAM_POOL region**: `0x34000000-0x341B1FFF` (1736K). The former SRAM APP space is now a general-purpose memory pool.
- **UNCACHED region adjusted**: Start address moved from `0x341A0000` to `0x341B2000`.

### 3. WiFi SDK Upgrade

- SiWG917 SDK upgraded to **v4.0.2 SDK**, bundled with WiFi firmware `SiWG917-B.2.15.5.0.0.2`.
- Older WiFi firmware (e.g., `2.14.5.x`) remains functional, but upgrading is recommended.
- New `EXPECTED_WIFI_VERSION` compile-time version check mechanism.

## New Features

### WiFi Firmware OTA Upgrade
- New OTA upgrade support for the SiWG917 WiFi firmware via the Web UI or system settings.
- **Two-stage upgrade flow**:
  1. **Upload stage**: Streaming write to Flash (`WIFI_FW_BASE`), writing `flash_header_t` + `.rps` firmware binary. The OTA header is used for Web-side validation only and is NOT written to Flash.
  2. **Confirm stage**: Sets NVS `wifi_mode=update` flag via `ota_upgrade_local_handler`. On next reboot, `wifi_mode_process()` → `firmware_upgrade_from_flash()` pushes the `.rps` to the SiWG917 chip.
- **WiFi version detection**: The system reads both the Flash-stored WiFi firmware version and the chip's current running version, displaying both on the firmware upgrade page with automatic mismatch detection.
- **Expected version check**: `EXPECTED_WIFI_VERSION` is set at compile time. When the Flash and running versions match but differ from the expected version, the Web UI shows an upgrade prompt dialog.
- **Upgrade waiting page**: Pushing the WiFi firmware to the chip takes ~1–3 minutes. The frontend polls the device after reboot and redirects to an LED-guided waiting page (`/upgrade-waiting`) on timeout.
- `firmware_upgrade_from_flash()` now uses `storage_lock/unlock` to prevent concurrent filesystem access conflicts, with proper resource cleanup on all error paths.
- New `wifi_mark_update_pending()` API: sets the NVS flag without immediate reset, allowing the HTTP response to return normally.
- New `wifi_get_running_version()` and `wifi_get_flash_version()` APIs: read WiFi firmware version from the chip and from Flash respectively.

### Web Recovery System
- New `web_recovery` module: automatically enters a built-in recovery page when the Web firmware partition is empty or corrupted.
- The recovery page is a self-contained HTML page compiled into the APP firmware (no external resource dependencies), providing Web firmware upload functionality.
- After a successful Web firmware upload, `web_recovery_reload_assets()` is called automatically to reload assets from Flash and exit recovery mode — **no device reboot required**.

### AI Partition Restructuring
- `FIRMWARE_DEFAULT_AI` renamed to `FIRMWARE_AI_2`, with `FIRMWARE_AI_1` retained for AI model A/B dual-slot switching.
- AI partition sizes increased from 5MB to 8MB to accommodate larger model files.

### Web Firmware Version Management Enhancements
- Firmware upgrade page overhaul:
  - Import firmware dialog supports APP / Web / AI Model / Device configuration files.
  - Advanced options menu provides dedicated FSBL upgrade and WiFi upgrade entry points.
  - New dedicated WiFi firmware upgrade page (`/import-wifi`).
  - Firmware info page displays all component versions: APP / Web / AI Model / FSBL / WakeCore / WiFi.
- **FSBL expected version check**: `EXPECTED_FSBL_VERSION` set at compile time; Web UI auto-detects mismatches and offers guided upgrade.
- **WiFi expected version check**: same mechanism via `EXPECTED_WIFI_VERSION`.

### HaLow (802.11ah) Support
- Full HaLow Wi-Fi integration: quick connect, runtime radio sync, reconnect IP fix.
- Extended RegDomain country code support (end-to-end).
- PKA crypto hardware offload.
- IP configuration persistence.
- SPI6 clock auto-adaptation, relaxed EU duty cycle.
- Web UI HaLow network configuration (halow-network.tsx).

### AI & Models
- YOLOv8-Nano / YOLO11n-Nano human pose estimation (4 keypoints).
- Pointer meter reading: YOLOv8-pose 4 keypoints → percentage reading conversion.
- Instance Segmentation (ISEG): YOLOv8 model integration.
- BlazeFace face detection: 128×128 resolution.
- Configurable AI inference cadence.
- Continuous AI result telemetry: CBOR compact format and metadata-only mode.
- SPE result handling: `overlay_results` gating.
- **Unified STEdgeAI 2.2 / 3.0 / 4.0 runtime selection at build time**: Application code is shared; select the STEdgeAI version via `STEDGEAI_VARIANT` (one of `2.2`, `3.0`, `4.0`). `stedgeai.mk` automatically picks the matching NPU runtime library (`NetworkRuntime1020/1100/1200`), device, and extra sources. Model release version is `$(STEDGEAI_BIT).$(MODEL_VERSION_OVERRIDE)`. Firmware and model packages must use the same variant. (commit `a69cf17`)
- **Generated `stedgeai_version.h`**: `MODEL_STEDGEAI_VERSION_SUPPORTED` is now a generated header (like `version.h` / `fsbl_version.h`) instead of a `-D` compiler flag. `nn.h` includes it so `-MMD` dependency tracking catches STEdgeAI variant changes — no more `make clean` required when switching variants.

### Capture-Upload Pipeline
- New capture-upload pipeline, with Web UI for scheduling capture tasks.
- Supports scheduled capture and manual trigger.

### Storage & File Management
- Web-based file management: browse, upload, and delete files on Flash and SD card.
- File operation registration decoupled from LFS mount for improved boot robustness.

### Hardware Support
- Auto-adaptive dual-chip NOR Flash driver: auto-detects MX66 (1Gb) and GD55 (1Gb/512Mb).
- 64MB and 128MB Flash capacity support (`BOARD_FLASH_SIZE`).
- ISP grayscale mode (reboot to apply).

### Networking
- 4G manual PLMN operator selection.
- eth_wan default interface handling.
- Modem init and power management fixes.

- **WiFi post-upgrade SDK init**: Added `sl_net_deinit` call after `firmware_upgrade_via_xmodem` completes to properly release the network stack.

## Improvements

- APP PSRAM boot is now the only mode (removed `BOOT_IN_PSRAM` conditional compilation); APP available space increased to 4096K.
- Flash SWAP region relocated from Flash to the start of the PSRAM region.
- CRC32 calculation switched from hardware CRC (`HAL_CRC_Calculate`) to software implementation (`CRC_Calculate`) to avoid hardware CRC peripheral endianness issues.
- Web firmware auto-reloads after OTA upload (no reboot needed), providing a smoother upgrade experience.
- Removed legacy si91x WiFi SDK code.
- **Build system — incremental build fix**: `write_if_changed()` in `version_header.py` only overwrites generated headers (`version.h`, `fsbl_version.h`, `stedgeai_version.h`) when content actually changes, preserving file timestamps so `.d`-based dependency checks don't trigger spurious rebuilds.
- **Build system — removed transitive `version.h` includes**: `#include "version.h"` moved from `core_init.h` and `json_config_internal.h` to the `.c` files that actually use version macros. This prevents 9 source files (`main.c`, `debug.c`, `driver_core.c`, `json_config_json.c`, `json_config_utils.c`, `quick_storage.c`, `webhook_service.c`, `api_webhook_module.c`, `api_ota_module.c`) from recompiling every time `version.h` changes.

## Known Issues

- WiFi firmware OTA upgrade push stage takes ~1–3 minutes. Do not power off during this process.
- Upgrading from v2.1.0 requires updating in sequence: APP → Web → FSBL → WiFi. See the [OTA Update Guide](./NE301_v2.3.0_OTA_Update_Guide.md).
